# Seam contracts — Phase 0 (static capture)

The externally-observable, durable seams that every later phase must preserve.
These are **captured statically** — from the WAD bytes and the legacy source, with
**no engine executed** — so they are a valid dark-regime data/seam baseline (see
`ORACLE_STRATEGY.md`). Each contract cites `../../ARCHITECTURE.md` and the
concrete `linuxdoom-1.10/` source.

> Rung: **L2-data / seam baseline**, *not* a behavioral golden master. The
> behavioral golden master (demo consistency checksum + frame hashes) is deferred
> to end of Phase 2 — see `ORACLE_STRATEGY.md`.

---

## 1. WAD lump format

Source: `linuxdoom-1.10/w_wad.h` (`wadinfo_t`, `filelump_t`);
`ARCHITECTURE.md` WAD deep-dive.

- **Header (12 bytes, little-endian):** `identification[4]` ("IWAD"/"PWAD"),
  `int numlumps`, `int infotableofs`.
- **Directory entry (16 bytes each):** `int filepos`, `int size`,
  `char name[8]` (NUL-padded, *not* NUL-terminated when 8 chars).
- Lumps are addressed **by directory index**; duplicate names are legal and
  resolved by scan order (`W_CheckNumForName` scans backwards). Order and
  duplicates are therefore load-bearing and preserved verbatim.

**Captured baseline for the pinned IWAD** (`wad-directory.tsv`, generated by
`tools/wad_oracle.py`):

| Field | Value |
|---|---|
| ident | `IWAD` |
| numlumps | `3163` |
| infotableofs | `28744468` |
| filesize | `28795076` |
| out-of-bounds lumps | `0` |

`wad-directory.tsv` pins every lump's `index, name, offset, size, crc32`. Any
change to lump content, size, offset, ordering, or duplicate structure changes a
CRC-32 and is caught by re-running the tool.

## 2. `ticcmd_t` — the determinism unit

Source: `linuxdoom-1.10/d_ticcmd.h`; `ARCHITECTURE.md` ADR + playsim deep-dive.

```c
typedef struct {
    char   forwardmove;  // *2048 for move
    char   sidemove;     // *2048 for move
    short  angleturn;    // <<16 for angle delta
    short  consistancy;  // net-game consistency check
    byte   chatchar;
    byte   buttons;
} ticcmd_t;
```

- 35 Hz fixed tic rate + the RNG table (§5) make a `ticcmd_t` stream fully
  reproducible — this is the engine's historical correctness oracle.
- `consistancy` is the in-stream desync detector for netgames/demos.

## 3. Demo LMP byte layout

Source: `linuxdoom-1.10/g_game.c` (`G_BeginRecording`, `G_ReadDemoTiccmd`,
`G_DoPlayDemo`); `DEMOMARKER 0x80` at `g_game.c:1488`.

- **Header = 13 bytes:** `VERSION, skill, episode, map, deathmatch, respawnparm,
  fastparm, nomonsters, consoleplayer`, then `MAXPLAYERS(=4)` × `playeringame`.
- **Body:** 4 bytes/tic — `forwardmove, sidemove, angleturn>>8, buttons`.
- **Terminator:** `0x80`.
- **Version gate:** `G_DoPlayDemo` rejects any demo whose first byte ≠
  `VERSION` (`= 110`, `linuxdoom-1.10/doomdef.h`).

**Captured baseline:** `demo-contract.md`. Critical finding — the pinned IWAD's
four demos (`DEMO1`–`DEMO4`) are **version 109** (vanilla DOOM 1.9) and are
therefore **rejected** by this engine (wants 110). Consequence: there is **no
replayable demo in Phase 0**; the demo-parity seed is **deferred to Phase 2** (see
`ORACLE_STRATEGY.md`).

## 4. Framebuffer + palette

Source: `linuxdoom-1.10/doomdef.h` (`SCREENWIDTH 320`, `SCREENHEIGHT 200`);
`linuxdoom-1.10/v_video.[ch]`; `ARCHITECTURE.md`.

- **Framebuffer:** 320×200, 8-bit paletted, one byte per pixel (palette index).
- **`PLAYPAL`:** 14 palettes × 256 colors × 3 bytes = **10752 bytes**. Pinned
  IWAD matches exactly (`wad_oracle.py --summary`).
- **`COLORMAP`:** 34 maps × 256 bytes = **8704 bytes** (32 light levels + invuln
  + 1). Pinned IWAD matches exactly.
- Frame hashes captured at Phase 2 will hash this 320×200 index buffer (and/or
  the palette-applied RGB) as the frame-smoke oracle.

## 5. RNG table + tic invariants

Source: `linuxdoom-1.10/m_random.c` (`rndtable[256]`, `P_Random`, `M_Random`);
`ARCHITECTURE.md` ADR.

- `rndtable[256]` is a **fixed compile-time constant** (value pinned by the
  source tree hash in `BASELINE_PROVENANCE.md`; first entries `0, 8, 109, 220,
  222, …`, last `120, 163, 236, 249`).
- `P_Random`/`M_Random` advance a single `& 0xff` index into the table — no
  entropy, no seeding. `P_Random` (playsim) determinism is the demo/net oracle;
  `M_Random` (cosmetic) must stay separate.
- **Tic rate:** 35 Hz fixed (`TICRATE`), the temporal quantum for all
  `ticcmd_t`-driven simulation.
- These are **WAD-independent** constants; they are frozen by pinning the source
  tree, not by a WAD CRC. Other WAD-independent tables (trig/tangent, gamma,
  state/sprite/sound definitions) are inventoried in `BASELINE_PROVENANCE.md`.

## 6. Savegame layout — **documented as intent, NOT empirically frozen**

Source: `linuxdoom-1.10/p_saveg.[ch]`; size/version handling in
`g_game.c` (~L1198–1312).

- The save system writes **raw C structs plus pointer swizzling and padding**.
  Struct sizes, alignment, and pointer widths are **32-bit-era assumptions**.
- On the modern **64-bit** target these will change; without a legacy save
  artifact captured from a running period binary, we can only document the
  *format intent*, not verify a byte-exact seam.
- **Status: high-risk / not empirically frozen.** Old-savegame compatibility is
  **deferred**; whether to preserve it at all is an explicit later decision (it is
  *not* a WAD/demo-style hard constraint). This matches
  `MODERNIZATION_PLAN.md` §8 ("savegame layout … preserved unchanged") only as an
  *aspiration* until an actual save artifact exists to diff against.
